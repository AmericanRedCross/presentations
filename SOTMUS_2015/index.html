maps always tell a story

Most of my job is about telling a story. To make that story we needed more data than what we can see from space.

friends

Red Cross and OSM are a match made in heaven, we started doing more with OSM in 2012 with a mapping project in Gulu, Uganda. OSM has awesome data, lots of amazing volunteers. We use OSM and work closely with HOT during disasters so we thought we could take that energy and put it into disaster prepardness.
maps certainly don't exist for most of the world's vulnerable population. OMK is part of the work that American Red Cross is doing around the world. Drishtie gave an awesome talk about missing maps earlier, if you didn't see it I encourage you to watch the video later.

making maps*

*and a little more

2 years ago at SOTMUS San Francisco I promised that Red Cross would become more and more involved with OSM and HOT. We've held up our end of the bargin. We have invested a lot of time and money into building workflows around OSM, creating data, and now creating OSM tools such as OMK and the OSM-Meta-Util.
We've also been doing mobile data collection pretty seriously for almost that entire time. Last year we transfered to a completely open stack from using proprietary tools. All parts of our international work involve mobile data collection. We've become experts in both OSM and ODK. Increasingly we saw the need to marry the two.

maps are mission critial when you don't know where you are

In my job I feel like I'm constantly lost looking for any data that I can get about wherever I am. OSM is the defacto source for humanitarians and the first place we look for data.

we need the *best* map

As we've worked with OSM we have constantly looked to see what else is out there. Does the tool that we need already exist, could we fund part of it, could we build it.

waiting in line sucks

Generally we like to be near the cutting edge of technology, never the bleeding edge. OMK is one instance where we weren't contenct to wait in line anymore. We needed the tool now.

just build it

So we decided to build it. We hired the awesome guys at SpatialDev in Seattle to build it for us.
Dhaka and Bangladesh is a strategic interest for the RC. Basically there are lots of people that need our help. Super high risk of disasters with an incredably vulnerable population.
World Bank and OpenCities did some work to create structured forms. Recording these still take lots of time to encode and push to OSM. We needed a tool that would do all of this in one phase. Decreasing the time from data collection to decision.
We've figured out the remote side of the workflow. Now we need to figure out the field data collection side.
Lots of mobile OSM tools exist but none met our structured use case.
OSM is great for us because we get to bring our own requirements and work with the community to create and manage data useful for them. Red Cross and other Humanitarians get to put our custom tags into OSM.
Most of the time this data freedom is nice until you get an email from the DWG telling you that all the work you did will be deleted because it doesn't conform to the 'spirit' of a tag
If you don't tell users exactly what you want you'll get back a hodepodge of data. This isn't very good for OSM and kinda terrible for the quality of our work.
OpenDataKit is all about getting people to answer questions in standard ways to derive the most information from them. Bad Data is really hard to analyze. We think about ODK survey answers like putting stuff into buckets. Good tagging in OSM is very similar, we don't versions of the same tag. We want the same tag.
First thing we did was to gather all the 'interested' parties together for a meeting in DC. We took all the requirements from everyone and tried to make it into a solid cohesive piece of work.

OMK bridge

OMK Bridge allows a user to parse the an osm file and submit edits one by one after doing some QA on the edits. Hope to integrate this with iD in the future. OMK is currently support by Ona and Ona Lite a free open source version of thier platform. Code is available on github
Bridge allows us to see which edits are outliers and resolve the edits. Its our attempt to see data in a first cut before moving it to something like JOSM.
OMK is an stand alone app and an extension of ODK similar to barcode or camera functionality. OMK is meant to be a seamless implementation.
The power of OMK is that it allows users to select the feature they are editing. This allows us
just like ODK, OMK is language agnostic. You can use any local language.
We have the ability to export all the edits at once within OMK Server
The average volunteer can collect between 100-200 builings a day.
part of our work flow is to pair field papers with OMK. This is very successful because we can then add new features to the map and still have the majority of the data collected in digital form.

10,000

To date we've added over 10K buildings to OSM using OMK in Harare, Zimbabwe, Chinhinda Rwanda, Mwanzi Tanzania, and Dhaka Bangladesh. Typical data included address, building materials, building quality, and builing use.
OMK is still in beta. The install process is a little complicated because we haven't pushed everything to do the play store yet. When this happens OMK will be available in the play store along with an updated version of ODK. We have successfully merged the changes to ODK that support OMK.
Our road map is pretty big there are lots of things we still need to do to OMK to be 100% happy with it. First up is pushing the current app to production and getting it to the play store. Followed by the ability to create POIs with a structured survey design, almost like PushPin but with the ability to create your own surveys.
